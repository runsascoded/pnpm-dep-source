# Vite resolve aliases for local deps

## Problem

When `pds l <dep>` creates a local symlink, Vite/Rollup can't properly resolve imports across the symlink boundary. This causes two classes of errors in any Vite project using `pds l`:

### 1. Duplicate React (hooks crash)

The local dep's `node_modules/react` takes precedence over the consumer's, causing two React instances at runtime. React hooks require a single instance â€” duplicates produce:

```
Cannot read properties of null (reading 'useRef')
```

This affects any local dep that has `react` as a peer dependency (pltly, use-kbd, use-prms, etc.).

### 2. Unresolved peer package imports

Dynamic imports in the symlinked dep can't resolve packages installed in the consumer. For example, pltly does `import('plotly.js-dist-min')` â€” Rollup tries to resolve this from pltly's directory, where `plotly.js-dist-min` isn't installed:

```
[vite]: Rollup failed to resolve import "plotly.js-dist-min" from "/path/to/pltly/dist/react/Plot.js"
```

This affects any local dep that dynamically imports a peer dependency.

### Current workaround

Manual `resolve.alias` entries in the consumer's `vite.config.ts`:

```ts
import { resolve } from 'path'

export default defineConfig({
  resolve: {
    alias: {
      'react': resolve(__dirname, 'node_modules/react'),
      'react-dom': resolve(__dirname, 'node_modules/react-dom'),
      'react/jsx-runtime': resolve(__dirname, 'node_modules/react/jsx-runtime'),
      'plotly.js-dist-min': resolve(__dirname, 'node_modules/plotly.js-dist-min'),
    },
  },
})
```

This must be done manually for every local dep's peer dependencies, and removed when switching back to `pds gh`/`pds n`. It's error-prone and non-obvious â€” users hit cryptic runtime errors with no clear connection to the local dep setup.

## Proposed solution

`pds l` should auto-manage Vite resolve aliases for local deps' peer dependencies.

### Approach A: Auto-patch `vite.config.ts`

When `pds l <dep>` runs:

1. Read the local dep's `package.json` â†’ `peerDependencies`
2. For each peer dep that exists in the consumer's `node_modules/`:
   - Add a `resolve.alias` entry pointing to `node_modules/<peer>`
   - For React specifically, also alias `react/jsx-runtime`
3. Write a managed block into `vite.config.ts` (or a separate file it imports)

When `pds gh <dep>` or `pds n <dep>` switches away from local:

1. Remove the managed aliases for that dep's peers
2. If no local deps remain, remove the managed block entirely

### Approach B: Generate a separate aliases file

`pds l` writes a `pds-aliases.ts` (or `.pds-vite.ts`) that the consumer's `vite.config.ts` imports:

```ts
// .pds-vite.ts (auto-generated by pds, gitignored)
import { resolve } from 'path'
const __dirname = import.meta.dirname
export const aliases = {
  'react': resolve(__dirname, 'node_modules/react'),
  'react-dom': resolve(__dirname, 'node_modules/react-dom'),
  'react/jsx-runtime': resolve(__dirname, 'node_modules/react/jsx-runtime'),
  'plotly.js-dist-min': resolve(__dirname, 'node_modules/plotly.js-dist-min'),
}
```

Consumer's `vite.config.ts`:

```ts
import { aliases } from './.pds-vite'

export default defineConfig({
  resolve: { alias: aliases },
})
```

This is less magical but requires a one-time setup in the consumer.

### Approach C: Vite plugin

`pds` ships a Vite plugin that reads `.pds.json`, checks which deps are local, resolves their peer deps, and auto-injects aliases:

```ts
import { pdsPlugin } from 'pnpm-dep-source/vite'

export default defineConfig({
  plugins: [react(), pdsPlugin()],
})
```

This is the cleanest DX â€” one-time plugin addition, then `pds l`/`pds gh` just works. The plugin reads `.pds.json` at dev server startup and build time.

## Implementation

**Approach C (Vite plugin)** was implemented in `src/vite.ts`, exported as `pnpm-dep-source/vite`:

```ts
import { pdsPlugin } from 'pnpm-dep-source/vite'

export default defineConfig({
  plugins: [react(), pdsPlugin()],
})
```

Additions beyond the original spec:
- `react-dom/client` sub-path alias (alongside `react/jsx-runtime`)
- `extra` option for additional modules to alias (e.g. dynamically-imported peers like `plotly.js-dist-min`)
- `root` option for non-cwd project roots
- Returns `undefined` (no-op) when no local deps or no `.pds.json`, so it's safe to leave in config permanently
- 8 unit tests covering all code paths

## Scope

This only affects Vite projects. Webpack has similar issues but different resolution mechanics. Start with Vite since that's the common case for pds users.

## Discovery context
